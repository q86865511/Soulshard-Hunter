Dev guide for working on this repo. Read this before editing. Keep it concise вҖ” **per-round detail lives ONLY in `docs/changelog/ROUNDx.md`** (the single home for version notes); this file describes the *current* architecture + the traps that bite.

## What it is
A single-player (optionally cloud-connected) Vampire-Survivors-style pixel roguelike. **Vanilla HTML5 Canvas + ES Modules, no build step**; the only external assets are the **recorded MP3 soundtrack** in `assets/music/`. All pixel art is procedurally generated in code; SFX are WebAudio-synthesised; music is the recorded soundtrack (state-mapped + crossfaded) with a procedural-synth fallback. Walk-only control вҖ” weapons auto-target and fire.

## Run / test
- **Serve:** `node tools/serve.mjs` вҶ’ http://localhost:5173 (a no-cache static server вҖ” do NOT use `python -m http.server`; it serves STALE ES modules after edits and silently breaks testing). `.claude/launch.json` exposes it as preview server "game".
- **Self-test in the browser:** `window.__DBG` exposes `snap()`, `reg()` (registry counts), `startRun()`, `autoplay(secs)`, `gallery()`, `enemyIds()`, `scene()`, `meta()`, `music()`, **`nav(name)`** (jump straight to `title`/`hub`/`run` вҖ” throttled rAF makes input-driven scene swaps unreliable), **`pump(n,dt)`** (manually tick N sim frames + one render), and the offline co-op self-tests **`coopRoundTrip()`** (encodeвҶ’decodeвҶ’render path), **`coopSilenceTest()`** (disconnect/retire), **`coopBossSyncTest()`** (R20 boss moves + 4 event mobs reconstruct on the guest).
- **Headless screenshots:** the preview tab is background-throttled, so `preview_screenshot` stalls. `tools/serve.mjs` exposes a dev-only `POST /__shot` that writes a base64 canvas capture to `_shot.png`; capture via `fetch('/__shot',{method:'POST',body:canvas.toDataURL('image/png')})` then Read `_shot.png`.
- **IMPORTANT вҖ” headless preview throttles `requestAnimationFrame`**, so `__DBG.autoplay` barely ticks and screenshots can be stale. Instead drive the sim manually from `preview_eval`:
  ```js
  const s = __DBG.scene(); s.player.takeDamage = () => {};      // godmode for inspection
  for (let i = 0; i < 9000; i++) { if (s.choice) s.choice = null; s.update(1/60); }
  ```
  Then read `s.run` / `s.world`. Call `s.render()` **sparsely** вҖ” it's expensive; rendering every tick can exceed the eval timeout. To screenshot a panel, set the field (`s.showBuild=true`, `s.bigMap=true`, a hub `s.panel='sortie'`) then call `s.render()` once.
- `window.__GAME_ERROR__` + the `#fatal` overlay surface uncaught errors.
- **Headless sim-test traps:** в‘  the first-battle `hudTut` walkthrough re-pauses the run ~2s in on a fresh save (`_hudTutShown` is per-scene) вҖ” clear `s.hudTut=false` INSIDE the pump loop or set `META.tutorialHUDDone=true`; since R21.8 the chapter `story` intro ALSO freezes the world for its first ~6.5s (solo only; skip by nulling `s.story` in the loop); в‘Ў anti-AFK drain writes `hp` directly (bypasses a `takeDamage` godmode stub) вҖ” a stationary godmode player still dies in ~4 min, so refill `s.player.hp` each frame; в‘ў `setScene()` queues вҖ” after calling it from `preview_eval`, run `__DBG.pump(1)` before `__DBG.scene()` or you read the OLD scene; в‘Ј also null `s.equipChoice`/`s.eventChoice`/`s.curseChoice`/`s.anvilChoice` in pump loops (any of them freezes `run.update`).
- After editing, **reload the page** before testing вҖ” `preview_eval` runs the in-memory module, not your new file.
- **Frontend smoke (CI gate):** `cd test && npm ci && npm run test:frontend` вҖ” Playwright headless Chromium over `tools/serve.mjs` (needs :5173 free); asserts boot, `reg()` counts, nav/render, the story pause, the codex/goals layer, and the co-op self-tests (incl. codex pollution guard). Runs in CI (`frontend-test.yml`) and gates deploy alongside server tests. **When a content round changes registry counts, update `REG_BASELINE` in `test/frontend-smoke.mjs` in the same PR.** Test deps live only in `test/` вҖ” the game runtime stays zero-dependency.
- **Server tests** (`server/` has `type:module`): `node server/test/smoke.mjs` (114) + `node server/test/social.smoke.mjs` (65). `node --check` works in `server/` but NOT on frontend ESM (see Gotchas). NOTE there are TWO fakepools вҖ” smoke.mjs's inline one AND `test/fakepool.mjs` (used by `npm run dev:fakedb`); a new SQL query needs a pattern stub in BOTH or dev mode 500s.

## Balance config (src/game/balance.js)
ALL difficulty / economy / pacing / status / map magic numbers live in `BALANCE` (one object) вҖ” tune there, not in gameplay code. Also exports `weaponMaxLevel`/`isWeaponMaxed` (weapons cap at **level 7**; character level is uncapped). Status effects live in `src/game/status.js` (`applyStatus`/`tickStatus`), enemyвҶ’player status tags in `content/status_tags.js`, achievement-gated unlocks in `content/unlocks.js`, the hidden dev cheat in `src/game/cheats.js` (Konami code вҶ’ in-run dev panel; `window.__CHEATS`).

## Core loop (run.js)
**File layout (R21.4):** `run.js` is a thin assembler вҖ” `export const runScene = Object.assign({}, вҖҰmixins)` over `src/game/scenes/run/*.js` (`shared.js` = file-scope helpers/constants; then `lifecycle`/`modes`/`events`/`combat`/`loop`/`coop`/`render_hud`/`shop_hidden`/`render`/`overlays` mixins). The scene is still ONE `this`-bound object (state lives on `this`); add new methods to the matching mixin, not back into run.js. The only duplicate key, `exit()`, lives in `lifecycle` (early) + `modes` (full) вҖ” `modes` wins via Object.assign order, as in the original literal.
A run = **one biome, 20 minutes** (`BALANCE.LEVEL_TIME`) at a chosen **difficulty** (`run.difficulty`/`diffMul`).
- **Threat** `this.threat` ramps 1вҶ’~13 over 20 min (`BALANCE.THREAT_PERIOD`) and drives enemy tier + scaling. Only **1-3 enemy types** active at once (`rotateTypes`, ranged biased down).
- **Mini-bosses:** a DISTINCT boss at 5/10/15 min (`miniBossTick`/`spawnMiniBoss`, `BALANCE.MINIBOSS_TIMES`), never the biome final boss, never repeated.
- **Hazards** = `world.hazards` (HAZ table, dmg Г—`TRAP_DMG_MULT`). **Special events** (`eventsTick`/`updateEvents`/`drawEvents`): mushrooms / surround ring / Higgs bombard + the 4 R20 event mobs (see Key subsystems), weighted via `BALANCE.EVENT_WEIGHTS`.
- **Status effects:** slow/bleed/burn/poison (DoT) + stun/knockup (CC) on player & enemies via `status.js`.
- **Finale:** at 20:00 the biome's `FINAL_BOSS` spawns; killing it вҶ’ `clearLevel` (unlock next biome + difficulty). 30s later (`REAPER_DELAY`) the killable **Reaper** descends; slay it or press E to leave. `finishRun` banks once. (з„ЎзӣЎ mode instead stacks curses past 20:00.)
- **Caps (no infinite pile-up):** weapons 6 / passives 14 (`MAX_WEAPONS`/`MAX_PASSIVES` in progression.js); weapon level 7; enemy hp/dmg + hazard/event damage scaling are clamped.
- Attack **tempo** ramps slowвҶ’fast (`world.playerTempo`/`enemyTempo`). Screen shake is dampened (`setShakeScale`, full only <25% HP).
- **Items/equip:** ground items auto-use on pickup (no inventory); equipment drops open a paused choose-to-equip menu (`equipChoice`). Anvils open a paused 3-choice (`anvilChoice`).

## Conventions when extending
- **Always add a changelog** вҖ” EVERY update/PR must add or extend a `docs/changelog/ROUNDx.md` entry (newest round) describing what changed + how it was verified. Don't scatter changelog `.md` files in the repo root; don't ship a change without one.
- **Content is data-driven** вҖ” register via `registry.js`. Never hard-code content in gameplay code; read from the registries. New magic numbers go in `BALANCE`.
- **New enemy:** `defineAnim('<sprite>', w, h, frames, (p,f)=>{ ...; p.outline(P.ink); }, {anchor:[ax,ay],fps})` then `Enemies.register({ id, sprite, ai:'chase|flyer|shooter|charger|wander', hp, speed, damage, radius, xp, gold, attack?, boss?, ... })`. Bosses (`boss:true`) auto-run the multi-phase system in enemy.js; wire data-driven moves via the `WIRE` map in `content/boss_moves.js` (NOT on the def вҖ” keeps gen defs re-integration-safe).
- **New weapon (auto-fire):** `Weapons.register({ id, icon:'weapon_<id>', tier, weight, maxLevel, cooldown(l), fire(world,p,inst), update?, draw?, levelDesc?, evolveInto?, evolveReq? })`. Evolves at maxLevel when the player owns `evolveReq` passive.
- **New character-exclusive weapon:** add a weapon-slot `Equipment.register({...,slot:'weapon',exclusive:true})` (hand-written `content/exclusives.js` uses an `X({...})` helper with a flat signature-weapon stat block) and map the owner in `CHAR_EXCLUSIVE`. Damage is applied at runtime via `makeEquipWeaponDef` вҖ” do NOT pre-multiply.
- **Icons:** `defineIcon('weapon_<id>'|'item_<id>'|'equip_<id>'|'ability_<id>', bgHex, (p)=>{...})`. **Character sprite:** `defineAnim('char_<id>',16,18,4,(p,f)=>{ drawHeroBody(p,f,id,art); p.outline(P.ink); },...)`.
- **Pixel API (`Painter`):** `px/rect/hline/vline/line/circle/ellipse/ring/mirrorX/outline/shadeBottom/replace`, plus polish helpers `glow/gradV/gradH/sparkle/star4/rimLight/softShadow/dither/speckle/aura` (rim light + soft shadow run AFTER the body, BEFORE `outline()`); shared palette `P` (anime accents `neon/magenta/sakura/aurora/astral/holy/rim` + ecosystem families `leaf/sand/bog/ocean/sky`), `sym` icon primitives. Missing sprites render a magenta placeholder (no crash); use `iconOr(name, fallback)`.
- **Stats** live in `makeBaseStats()` (state.js). Only write stat fields that exist there, or you'll get `NaN`.
- Equipment slots: `armor`/`trinket` apply stat mods; `weapon`-slot equips become a real **signature auto-weapon** via `makeEquipWeaponDef` (equipItem). Weapon-slot equips are sold in the in-run shop, not dropped.

## Workflow content pipeline
Content was mass-produced by multi-agent Workflows (generate вҶ’ adversarial review). A workflow returns `{result:{packs:[{key,code,ids,spriteNames,ok}]}}`; normalize to `{result:packs}` then run **`node tools/integrate.mjs <file.json>`** вҖ” it decodes HTML entities, prepends the import header, writes `src/art/gen/*` + `src/game/content/gen/*`, and rebuilds the fault-isolated dynamic-import indexes by scanning the dirs. Re-run to integrate new output. **Gen files are overwritten on re-integration** (see Gotchas).

## Gotchas
- `KeyM` = **minimap**, `KeyB` = **shop** (not mute вҖ” mute lives in the settings menu). `Tab` = build page. `Shift` = dash (rebindable). Mouse `wheel` is wired (`mouse.wheel`); hub long panels have a draggable scrollbar.
- Save: key `soulshard.save.v1`, `SAVE_VERSION` + migration in `loadMeta`; reset backs up to `вҖҰv1.bak`.
- `node --check <file>` only checks syntax (won't resolve imports) вҖ” and treats the no-`package.json` frontend `.js` as CJS, so it **errors on `import`/`export`**; for frontend ESM, validate by reloading + the manual-pump test. (`server/` has `type:module`, so `node --check` works there.)
- **Gen-file edits get reverted by re-integration.** `src/game/content/gen/*` + `src/art/gen/*` are workflow-generated; several have been hand-edited (gen_weapons_a/b/c `BALANCE`/`PLAYER_DAMAGE_MULT`/crit/forge-swift fixes; `gen/talents.js`+`gen/facilities.js` cost MULs + value trims; `gen_items_anvils.js` diminishing returns; halved gold sources in a few gen ability/gear files). Re-running `integrate.mjs` overwrites them вҖ” re-apply, or patch the workflow source. Hand-written content files (exclusives, heroes_r20, weapons_r20, boss_moves, event_mobs, bonds, the co-op netcode, вҖҰ) are NOT touched by re-integration.
- **Sprite bake is EAGER** (`defineAnim` runs the draw fn + caches frames at definition time). Any art a sprite depends on (e.g. `HERO_ART` bodies) must be registered BEFORE that sprite is defined. `characters.js` self-imports `art/heroes.js` so `HERO_ART` is populated before its eager bake loop; `state.js` transitively imports `content/characters.js` very early in boot, so character sprites bake before main.js's later art imports unless the dep is made explicit.
- Forge/grid rows that stash a non-rect payload in `r.w` must hit-test with a separate width field (e.g. `r.wd`) вҖ” `inside(mx,my,r)` reads `r.w` AS the width.
- **Forge з–ҫйҖҹ/swift covers update-driven weapons:** `player.updateWeapons` stashes the effective haste on `inst.fmHaste` before calling `update()`; each update-driven weapon (beams/auras/turrets) divides its self-gating re-arm timers and per-enemy hit cooldowns by `(inst.fmHaste || 1)`.
- **Enemy proximity queries use a uniform spatial grid:** `world.rebuildGrid()` runs once per update; `world.forEachNear(x,y,r,cb)` replaces O(nВІ) scans in enemy separation, `nearestEnemy`, `dealAreaDamage`, and projectileвҶ’enemy combat. Grid cell = `TS*4`; clean at the 260-enemy cap.
- **Co-op:** `Player.maxHp` is a **getter** (`return this.stats.maxHp`) вҖ” never assign it; the guest stores the networked max as `pl.nmax`. `world.player` stays the LOCAL avatar; `world.players` is the full set. The WS path has its OWN rate limiter (`realtime._allow`) вҖ” `@fastify/rate-limit` only guards HTTP. The enemy tuple's last field is `mvLift` (t[8]); guest `makeEnemy` inits it to 0 and `applySnapshot` resets it each frame so the leap_slam lift releases cleanly.
- **Co-op testing:** a backgrounded/headless tab queues WebSocket `onmessage` until JS next runs вҖ” so `RT.room` lags between `preview_eval` calls; poll `RT.room.members.length` inside an active eval before asserting. Run only ONE dev server on :8787 (kill zombies via `taskkill //F //PID <netstat-pid>`). Prefer the offline self-tests `__DBG.coopRoundTrip()` / `coopSilenceTest()` / `coopBossSyncTest()` вҖ” they exercise the encodeвҶ’decodeвҶ’render path without two real tabs or a relay server.
