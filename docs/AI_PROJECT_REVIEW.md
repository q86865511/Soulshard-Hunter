# Soulshard Hunter 專案體驗、玩法與工程評估

> 評估基準：`main` commit `03af175c08982135ea1cd0f2aa2a949a3dfaa111`（2026-06-29）<br>
> 評估日期：2026-07-10<br>
> 對象：下一輪負責規劃與實作的 Claude Code<br>
> 性質：產品／遊戲設計／UX／工程綜合審查，不是完成長期玩家研究後的最終平衡結論。

## 1. 結論先行

Soulshard Hunter 已不是「缺功能的原型」，而是一個內容與工程骨架都相當完整的可發布作品：純 Canvas 程序化像素美術有一致辨識度，單機離線優先，局內 Build、城鎮長期成長、每日／週常／無盡、雲端與 1～3 人合作均已有實作；後端也具備測試、部署、權威計分與管理工具。

下一階段最大的報酬不在繼續大量增加角色、武器或城鎮系統，而在「讓玩家更快理解現有內容、讓第一局更安全流暢、用資料判斷平衡、把前端回歸納入 CI」。目前的主要風險是內容密度與呈現順序已超過新玩家能一次吸收的程度，而前端最關鍵的遊戲流程仍主要依賴人工／瀏覽器內自測。

建議下一輪遵循以下順序：

1. 先修正第一局教學期間模擬仍運行的問題，並讓部署前會自動驗證前端核心流程。
2. 短期明確標示桌面鍵盤優先；不要把目前僅能縮放、不能完整移動的手機版視為正式支援。
3. 再改善內容發現、Build 教學、無障礙選項與可觀測性。
4. 短局模式、手動瞄準或完整手機操作應視為需要數據／真人測試驗證的獨立里程碑，不與第一批修正混做。

## 2. 評估方法與證據等級

### 已驗證事實

- 閱讀 `README.md`、`CLAUDE.md`、`docs/TEST_REPORT_R21.md`、`docs/changelog/ROUND21.md`、CI workflow、主要輸入／設定／戰鬥迴圈／進度／資料庫程式碼。
- 內容規模由目前文件與 registry 自測紀錄交叉確認：27 角色、43 武器、54 被動、60 裝備、63 敵人、10 生態、219 成就。
- 正式站以 1280×720 桌面視窗走過：載入、標題、空存檔、首段城鎮對話、出擊面板、第一局章節文字、HUD 教學、戰鬥、第一次升級三選一、暫停與設定。
- 首次升級約在本次短測第 24 秒出現；三張卡片的圖示、類型、名稱與效果在桌面尺寸可辨識。
- 第一局章節文字顯示時，畫面計時由約 0:02 繼續前進至 0:06；程式上 `this.run.time += dt` 與 `world.update(dt)` 也位於 `this.story` 處理之前（`src/game/scenes/run/loop.js`）。
- 390×844 手機直向視窗會重排 Canvas 與設定面板，但 `touchstart/touchmove` 只更新 `mouse`（`src/engine/input.js`），實際角色移動仍由 `moveAxis()` 的鍵盤集合驅動（`src/game/player.js`、hub／coop scene）。
- 正式站短測期間 `window.__GAME_ERROR__` 為空，瀏覽器 console 無 warning／error。
- GitHub Actions 的 CI 與 deploy gate 只呼叫 `.github/workflows/server-test.yml`；前端 `__DBG` 測試沒有進 CI。
- `server/README.md` 仍描述 18＋33 項測試，但 `CLAUDE.md`、workflow 與最新驗證紀錄是 smoke 103＋social 65＝168。
- 後端 `runs` table 保存最終對局摘要；管理統計目前以帳號、對局數、線上人數與排行榜聚合為主，沒有首局步驟、死亡原因、選卡或 Build 組成事件。

### 合理推論

- 連續的存檔選擇、城鎮對話、出擊指南、章節文字與 HUD 導覽會增加首局認知負荷。這是依實際流程與資訊層數推論，仍應用 5～10 位首次玩家測試確認卡點。
- 內容量已足以支持長期遊玩，但若缺少「下一個可達成目標」與「已發現內容」整理，玩家可能把深度感受成雜訊或 grind。
- 20 分鐘標準局對核心玩家合理，但會提高首次失敗與短時間使用情境的重試成本；是否需要短局模式必須先看退出時間分布。

### 本次未驗證

- 未登入帳號、未上傳排行榜、未操作管理後台。
- 未做兩名真人、真實網路延遲下的合作遊玩；多人功能判斷主要依程式碼、既有自測與 R21 測試報告。
- 未完成 20 分鐘通關、死神戰、無盡／每日／週常完整局。
- 未進行控制器、觸控真人操作、低階裝置效能、色覺或動態敏感族群測試。
- 未取得留存、退出點、Build 勝率等玩家行為資料，因此平衡與重玩性評分是設計審查，不是統計結論。

## 3. 專案優勢

### 明確且一致的作品辨識度

- 除實錄配樂外，像素美術由共用 Painter API 程序化生成；角色、敵人、生態與城鎮維持一致的視覺語言。
- 標題畫面、城鎮與局內不是單純換色：正式站短測可明顯辨識場景層級、互動點與像素動畫。
- 12 首狀態配樂加合成器 fallback，使純前端架構仍有完整音樂層次。

### 核心循環已成立

- 走位＋自動攻擊降低輸入門檻，第一次升級很快出現，Build 決策能在首分鐘開始。
- 6 武器、14 被動、進化／融合、12 羈絆、角色專武、稀有度、裝備與局內事件提供足夠的組合空間。
- 小王、事件、最終 Boss、死神、隱藏房、寶庫與地圖互動物，讓 20 分鐘不只是固定怪潮。

### 長期成長與重玩鉤子完整

- 10 生態、D1～D5、無盡、每日、週常與獨立排行榜已涵蓋不同回流理由。
- 可走動城鎮不是選單包裝：鍛造、天賦、設施、公會、銀行、衣帽、NPC 好感、寵物與房間裝飾建立持續世界感。
- 成就同時作為目標與內容解鎖，角色也有免費、金幣與成就等不同取得路徑。

### 工程基礎優於一般個人作品

- 純 ES Modules、零前端 runtime dependency、資料驅動 registry、集中式 `BALANCE`，新增內容的接縫清楚。
- 120 Hz 固定步、uniform spatial grid、敵人上限與倍率 cap 表明專案曾實際處理效能與數值失控問題。
- 離線優先、本機存檔 fallback、雲端存檔、伺服器重算分數與主機權威合作的責任邊界明確。
- 後端 168 項 smoke/social 測試、部署前 gate、Fastify v5 安全升級與可重入 schema，維運成熟度高。

## 6. 建議詳述與驗收

### P0-1：第一局教學必須真正暫停

**問題與玩家影響**
第一局章節文字是玩家理解世界觀與目標的時刻，但目前 `run.time`、`world.update`、生成與傷害都可能在其背後繼續。玩家可能在閱讀時受傷，甚至把教學誤認為遊戲不公平；同時教學耗時會污染首局計時。

**證據**
`src/game/scenes/run/loop.js` 先更新 `run.time` 與世界，之後才處理 `this.story`；正式站短測的畫面計時亦持續前進。相較之下 `hudTut` 已在 update 前段 early return，會凍結戰場。

**建議方案**
將 story 與 HUD walkthrough 都視為 blocking onboarding state：只更新教學自身的倒數／輸入，不更新世界、局內計時、生成、事件與 AFK。關閉後下一幀正常恢復。保持空白鍵必須放開再按的既有防誤跳設計。

**驗收條件**

- fresh save 進入首局，story 存在時 pump 180 simulation frames，`run.time`、玩家 HP、擊殺數與敵人數不因戰鬥更新改變。
- story 自然結束與按空白跳過兩條路徑都能恢復戰鬥。
- HUD walkthrough 仍會暫停，且不會重複彈出。
- 非首局、每日、無盡、單機／host 路徑沒有新的停滯。

**不做什麼**
不重寫劇情、不新增教學頁、不在這批調整敵人數值。

### P0-2：前端測試進 CI

**問題與玩家影響**
後端有強測試與部署 gate，但遊戲本體的 boot、scene import、registry、Canvas render 與 co-op encode→decode 只靠人工或未提交的 scratch Playwright。任何遺漏 import、生成內容錯誤或教學 state regression 都可能在 server test 全綠時部署。

**建議方案**
新增獨立的 dev/test-only Chromium 測試依賴，不改變前端零 runtime dependency：啟動 `tools/serve.mjs`，以 headless browser 執行 boot、registry 計數、title→hub→run、教學暫停、`coopRoundTrip()`、`coopBossSyncTest()` 與 `__GAME_ERROR__` 斷言。建立可重用 frontend workflow，讓 PR CI 與 deploy 同時依賴 server＋frontend 測試。

**驗收條件**

- 測試在 fresh checkout 的 GitHub-hosted runner 可重現，不依賴本機存檔、Postgres 或正式站。
- 任一關鍵 module import 錯誤、`__GAME_ERROR__`、registry 計數錯誤或 story 暫停 regression 都會使 workflow 紅燈。
- 測試不改 repo tracked snapshot，不依賴固定 sleep 判斷遊戲結果。
- deploy job 必須同時等候 server 與 frontend gate。

**不做什麼**
不把所有 UI 冷路徑一次自動化；先保護最關鍵的可發布路徑。

### P0-3：平台支援必須誠實

**問題與玩家影響**
Canvas 能在手機直向縮放，`input.js` 也有「mobile fallback」註解，容易讓人認為手機可玩；但 touch 沒有形成角色移動向量，也沒有衝刺、互動、選卡與暫停的完整觸控層。

**建議方案**
本批先採桌面優先：偵測窄視窗或 coarse pointer，在標題頁顯示非阻擋提示「目前建議使用實體鍵盤遊玩」，README 操作區明確列出正式支援輸入。完整手機操作另列 P1，不用零散點擊 fallback 繼續堆疊。

**驗收條件**

- 390×844 與 coarse pointer 會看到提示；1280×720 桌面不顯示。
- 提示不遮住單人／多人主按鈕，不宣稱手機正式支援。
- README 與遊戲內文字一致。

**不做什麼**
不在 P0 實作虛擬搖桿或改變桌面按鍵。

### P1-1：把「大量內容」轉成「可理解的目標」

**問題與玩家影響**
遊戲已有 27 角色、43 武器、54 被動、60 裝備與 219 成就；繼續增加內容的邊際價值正在下降。玩家更需要知道「這局可以追什麼」「剛剛解鎖了什麼」「融合曾如何成功」。

**建議方案**

- 城鎮提供 1～3 個依當前進度產生的推薦目標，而非一次展示所有系統。
- 建立內容圖鑑：未發現顯示剪影／提示；取得過的武器、被動、羈絆與 Boss 顯示完整說明。
- 保留首次融合的探索感，但成功後永久記錄配方；升級三選一可標示「與已持有項目有已發現配方」。
- 成就預設顯示「接近完成／有實質解鎖獎勵」，完整 219 項仍可切換查看。

**驗收條件**

- fresh save 不會因新圖鑑暴露隱藏內容；舊存檔載入不丟資料。
- 玩家能在兩次操作內找到下一個角色／武器／生態解鎖條件。
- 已完成一次的融合配方會跨局保存且可搜尋。

**不做什麼**
不刪除既有內容、不直接公開所有秘密、不新增另一種貨幣。

### P1-2：無障礙與難度個人化

**問題與玩家影響**
現有設定包含音量、震動開關、靜音、UI 縮放與按鍵重綁，是良好起點；但大量粒子、螢幕震動、狀態色與怪潮密度會排除動態敏感、低視力或色覺不同的玩家。

**精簡競品基準**
《Brotato》官方頁明列可調整敵人生命、傷害與速度，並把它視為 accessibility；其短波次與局間商店也降低單次失敗成本。《Vampire Survivors》則以極簡操作與局內金幣帶回局外升級，讓核心循環非常容易說清楚。Soulshard Hunter 不必照搬，但應學習「讓深度存在於選擇，而不是存在於理解介面的成本」。

來源：[Vampire Survivors Steam](https://store.steampowered.com/app/1794680/Vampire_Survivors/) · [Brotato Steam](https://store.steampowered.com/app/1942280/Brotato/)

**建議方案**

- 將畫面震動從 bool 改為 0～100%，加入螢幕閃光／粒子密度／傷害數字開關。
- 危險預警同時使用形狀、輪廓與動畫，不只靠顏色。
- 在劇情或自訂輔助模式提供敵人生命、傷害、速度倍率；清楚標示是否計入排行榜，避免破壞競賽公平。

**驗收條件**

- 全部選項可即時預覽、存檔、重載後保持。
- 輔助倍率不影響標準排行榜，單機離線仍可用。
- 關閉粒子／閃光後，Boss 預警仍可辨識。

**不做什麼**
不把無障礙倍率偷偷套進標準排行榜，不在本批重做全部戰鬥美術。

### P1-3：建立能回答產品問題的遙測

**問題與玩家影響**
目前最終 run summary 可以做排行榜與基本統計，但回答不了「多少人進入第一局」「在哪個分鐘退出／死亡」「哪些卡從不被選」「哪些 Build 通關率異常」。缺少這些資料時，平衡容易依少量自測反覆調整。

**建議方案**
先定義最小、隱私友善且可停用的事件：`save_created`、`tutorial_step`、`run_started`、`level_choice`、`run_ended`、`unlock_seen`。run end 補死亡來源、武器／被動 IDs、角色等級與版本。事件採白名單 schema、批次送出、不收自由文字與裝置識別碼；管理後台只做聚合。

**驗收條件**

- 可計算首次啟動→出擊→3 分鐘→首局結束的漏斗。
- 可依版本／角色／生態／難度查看通關率、死亡時間與選卡率。
- 離線時不阻擋遊戲；登入不是必要條件；玩家可停用分析。
- 明訂資料保存期限與不收集欄位。

**不做什麼**
不在資料庫保存逐幀位置、不收聊天／輸入內容、不以遙測取代真人訪談。

### P1-4：讓結算教玩家下一局怎麼變強

**建議方案**
沿用既有 `dmgBySource` 與結果排行，新增死亡來源、承傷最高來源、Build 關鍵協同、未完成融合提示，以及一條具體建議。例如：「主要傷害來自魂晶彈；已滿級但缺少融合條件」「近戰接觸造成 62% 承傷，可優先選移速／擊退」。

**驗收條件**

- 建議由實際 run data 產生，不使用與本局無關的隨機文案。
- 勝利與死亡皆有正向摘要；無法判斷時不顯示武斷建議。
- 結算不新增需要玩家管理的永久資源。

**不做什麼**
不以單一死亡來源武斷判定玩家技術，不新增會懲罰失敗的局外扣款。

## 7. 三階段 Roadmap

### 立即修正：發布可信度與第一局安全

1. story／HUD onboarding blocking state 回歸測試。
2. 修正教學期間世界與計時繼續的問題。
3. 前端 smoke 加入 CI 與 deploy gate。
4. 手機顯示桌面優先提示、修正 README；更新 server test 數量。

完成標準：fresh save 的第一局不會在閱讀教學時受傷；前端 boot／scene／registry／教學／co-op 回歸任一失敗都不能部署。

### 下一里程碑：讓既有深度更容易被看見

1. 內容圖鑑、已發現融合配方、推薦目標。
2. 成就預設聚焦接近完成與有解鎖獎勵項目。
3. 無障礙設定與危險提示強化。
4. 結算教練與最低限度產品遙測。

完成標準：首次玩家可以說明下一個目標；回鍋玩家能找到新內容；開發者能以資料回答首局與 Build 平衡問題。

### 需數據驗證的實驗

1. 若大量玩家在 5～12 分鐘離開，製作獨立的 8～12 分鐘快速遠征；不要直接縮放標準局全部數值。
2. 若玩家反映自動瞄準缺少掌控感，原型測試方向鎖定／可選手動瞄準。
3. 若手機流量或需求足夠，再投入完整 virtual controls 與裝置矩陣測試。
4. 以兩地真人合作測試決定是否優先改邀請、延遲提示、主機遷移或訪客升級介面。

## 8. 給後續 AI 的決策原則

- 預設先打磨，而不是再進行一輪大規模內容生成。
- 每個功能必須回答：改善哪個玩家問題、如何觀測、什麼條件代表成功。
- 不把「已有很多系統」等同於「玩家理解這些系統」。
- 保留離線優先、零前端 runtime dependency、資料驅動 registry 與單機路徑穩定性。
- 高風險的平衡、多人與資料庫變更要分批，不能跟純 UI polish 混成一個無法回歸的大 patch。
- 沒有真人長局或遙測證據的項目，標示為實驗，不寫成已證實問題。
